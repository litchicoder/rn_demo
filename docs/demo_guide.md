# React Native 核心原理：现场演示指南 (Live Demo Guide)

> [!IMPORTANT]
> 本指南旨在通过实时工具，将“看不到”的底层原理（Bridge, Virtual DOM, Yoga, Fabric）可视化。

---

## 演示前准备 (Checklist)
> [!TIP]
> 演示确认以下几点：
> 1. 确保演示设备（手机/模拟器）与开发环境网络能够互相通信。
> 2. 提前开启 **React DevTools** 或桌面端 **Flipper**，确保能正常抓取渲染树。
> 3. 预先写好触发长耗时（如百万次循环计算）的测试按钮代码。

## 演示一：虚拟 DOM 的“找茬”游戏 (Highlight Updates)
**展示目标**：展示 React 的局部更新能力，证明并没有全量刷新 UI。

### 操作步骤
1. 进入应用内的 **V-DOM 找茬** 页面。
2. 该页面已经内置了 **绿框闪烁特效 (Render Visualizer)**。任何组件只要触发了 `render`，周围就会闪烁一次绿框。
3. **现场观察**：
   - 屏幕上只有一个处于自动计时的组件周围在**每秒闪烁**，而这并未导致整个屏幕重新渲染（外层边框不闪烁）。
   - 点击“触发父组件更新”按钮，整屏、以及普通子组件会闪烁（这就是级联渲染效应），但由 `React.memo` 保护的组件**绝对不会闪烁**。
   - **讲解核心**：告诉观众，虽然外围或者定时器的 JS 在重新运行，但 **Virtual DOM** 对比后发现界面上其他 DOM 节点没有变化，所以最后它只通知原生端精准地更新了变动的地方。这是 React 高效的根本！

## 演示二：Yoga 的布局魔法 (Inspector)
**展示目标**：展示样式坐标转换。你可以写 Flexbox，但 iOS/Android 获取的是具体的 X, Y 坐标。

### 操作步骤
1. 打开电脑端的 **Flipper (Layout 插件)** 或 **React DevTools (Elements 面板)**（大屏幕操作视觉冲击力更强）。当然，也可以摇晃手机调出系统自带的 **“Show Inspector”**。
2. 开启组件检查模式，点击屏幕上的任意 UI 元素。
3. **现场观察**：
   - 此时可以把鼠标悬浮在组件树上，让观众看到屏幕上精准框出的真实像素边界。
   - 查看 “Box Model” 或布局属性面板。
   - **讲解核心**：在这里你能看到真实的 Margin, Padding 等精确到小数的坐标值。强调 **Yoga 引擎就像一个“同声传译”**，瞬间把你写的网页端特性 `flex`, `justifyContent`, 算成了 iOS/Android 底层 UI 系统能听懂的具体 `(x, y, width, height)` 屏幕绝对物理尺寸！

## 演示三：流畅度大考验 (Fabric 引擎 vs 老 Bridge)
**展示目标**：通过人为锁死 JS 线程，直观对比老 Bridge 架构的致命缺陷与新 Fabric 引擎 (JSI) 的破局机制。

### 操作步骤
1. 开发者菜单中点击 **“Show Perf Monitor”**。
2. 屏幕上会出现一个小窗口，时刻关注 **JS FPS** 和 **UI FPS** 的变化。
3. **现场演示与观察**：
   - 点击“阻塞 JS 3 秒”按钮，触发一段同步的 `while` 死循环代码。
   - **视觉震撼**：你会发现 **JS FPS 瞬间跌至 0（完全假死）**，但此时去滑动屏幕上的 ScrollView 列表或观察旋转动画，它**依然保持 60帧 的丝滑（UI 不卡）**。

### 🎤 现场讲解核心 (Fabric 视角解析)

> [!CAUTION]
> **抛出问题（反转）**：很多开发者以为“JS 卡 UI 不卡”是好事，但这就掩盖了老架构最大的痛点！

1. **为什么 UI 不卡死？（跨端多线程的共性）**
   - 无论是老架构还是 Fabric，原生 UI（滑动、原生动画）都在独立的 Native Main Thread 运作。此时 JS 被 `while` 锁死，原生层依旧能照常接管手势并渲染动画。

2. **老 Bridge 架构的灾难（反面教材）**
   - **异步黑盒排队**：在这 3 秒 JS 卡死期间，如果遇到必须由 JS 计算的交互（如手指点击点赞变色）。老架构下，用户的交互事件会被序列化成 JSON 字符串塞入 `Bridge` 异步队列。
   - **报复性卡顿**：由于 JS 陷入死循环没人取消息，3 秒后 JS 恢复，瞬间面临成百上千条滞后的 JSON 积压消息。页面将出现极其严重的 **“跳帧”、“操作延迟”乃至状态错乱**。

3. **Fabric 渲染引擎的终极破局（真正的重点）**
   - **C++ Shadow Tree 接管**：Fabric 用 C++ 重写了组件树，布局数据早已在底层准备就绪，不需要跨端来回索要。
   - **干掉 JSON，拥抱 JSI 直连**：Fabric 彻底废弃了笨重序列化的 JSON Bridge，改用 **JSI (JavaScript Interface)** 让 C++ 与 JS 对象直接互相持有内存级指针调用。
   - **同步渲染突围**：一旦发生高优先级交互，Fabric 结合 React Concurrent 模式，利用 JSI **在当前帧内同步穿透到原生底层**完成 UI 布局计算与重绘，彻底绝杀了“消息排队等几帧”的尴尬白屏！

4. **⚡️ 进阶拷问：3 秒锁死结束后，积压的交互事件去哪了？**
   - **痛点前提**：如果是纯 `while` 死循环，无论新老架构 JS 都绝对无法执行（假死）。真正的差距体现在 **3 秒恢复瞬间的“擦屁股（处理积压）”能力**。
   - **老 Bridge 灾难现场 (多张底片重影曝光)**：恢复后，JS 逐个处理排队的 JSON 事件。结果就是画面像走马灯一样连续疯狂跳帧、闪烁，触发多轮昂贵的序列化渲染。
   - **Fabric 的优雅收场 (一步到位冲印成片)**：恢复后，底层 JSI 积压的事件会触发 React 18 的**自动批处理 (Automatic Batching)**。React 会智能折叠中间状态，一口气算出最终结果，随后利用 JSI **“一步到位、同步且只触发一次”** 地更新 C++ 渲染树。没有闪烁，干净利落！
